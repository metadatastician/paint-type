== What this is, and how it stays trustworthy

*What it is.* A short, dated, signed snapshot of what we can _honestly and
verifiably_ claim about *paint.type* at one exact commit. Marketing and promises
live in the README. This file is the receipt.

*How it stays trustworthy.* Three moving parts: (1) *Ground truth, not memory* —
every claim below was produced by _running the project's own checks_ in the
session that wrote this file (`idris2 --check`, `cargo fmt --check`,
`cargo test`) on 2026-06-29; where a status doc, a CI badge, or memory disagrees,
the live run wins and the contradiction is flagged. (2) *A frozen anchor* — the
exact SHA, branch, UTC timestamp, and toolchain (see <<verifiable-anchor>>).
(3) *A real signature* — landed by a signed git commit, so the affirmation is
tamper-evident and attributable.

*We are fallible.* This is our best honest belief, not a proof of its own
correctness. Treat it as a falsifiable claim, not gospel.

== The epistemic contract (read before you trust _or_ attack)

This document records our *best belief* at the timestamp below. The only
guarantee is *no intentional overclaim*: where something checks we say
"affirmed"; where it is aspiration, an unbuilt milestone, unpushed work, or a

later turns out false is an *error to be fixed* — not a lie. You are invited to

== What we do NOT claim — and live contradictions found

*The Rust core does NOT link on `main`.* `src/paint_core/src/lib.rs` declares
twelve `pt_layer_*` `extern "C"` functions, but the libpt source on `origin/main`
(`src/interface/ffi/`) exports *none* of them — only the `pt_tile_*` /
`pt_alloc_u` / `pt_version` set. So `cargo test` for `paint_core` *cannot link*
on `main` (on CI or locally): `undefined symbol: pt_layer_push`, etc. This is
currently *masked* because `rust.yml`'s check/fmt job fails first and the test
job is **skipped**. *We do not claim `paint_core` builds or tests green on
`main`.* The linkage fix exists only on an *unpushed* local branch
(`16db486`, "resolve libpt layer FFI linkage"), entangled with unfinished v0.3
work — see below.

*The published Rust test counts could not be reproduced on `main` this session.*
The README/ROADMAP "98/98" (now ~104) figures correspond to the unpushed branch
built with the *pinned* zig 0.15.1. On `origin/main` with the available zig
0.15.2, the link fails (the `pt_layer_*` gap above, plus a `__zig_probe_stack`
toolchain skew between local 0.15.2 and pinned 0.15.1). *Reported by docs/CI,
not re-verified here.*

*CI on `main` is red.* `rust.yml` fails at `cargo fmt --check` (this PR clears
that) and would then fail at the `pt_layer_*` link; `idris-ci` was stale-red
until the Platform.idr fix in this change; OpenSSF Scorecard reports
`startup_failure`. The README's green CI badges currently overclaim.

*The v0.3 desktop-shell backend is unfinished WIP* (not on `main`, not in this
PR). `src/backends/cpu/main.zig` is mid-migration to the zig-0.15 `ArrayList`
API (`Layer.deinit`, `ArrayList.init` errors) and does not compile.

*No runnable application exists.* paint.type is pre-alpha; there is nothing an
end user can run today. The launcher's startup target is not built.

*Everything in README "What it will be" is aspiration* — layers UI, plugin
browser/SDK/sandbox, real-time Burble collaboration, LLM channel, colour
management. None are built.
