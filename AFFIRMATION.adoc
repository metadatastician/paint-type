== What this is, and how it stays trustworthy

*What it is.* A short, dated, signed snapshot of what we can _honestly and
verifiably_ claim about *paint.type* at one exact commit. Marketing and promises
live in the README. This file is the receipt.

*How it stays trustworthy.* Three moving parts: (1) *Ground truth, not memory* —
every claim below was produced by _running the project's own checks_, or by
reading live CI results, in the session that wrote this file, on 2026-07-04;
where a status doc or memory disagrees, the live result wins. (2) *A frozen
anchor* — the exact SHA, branch, UTC timestamp, and toolchain (see
<<verifiable-anchor>>). (3) *A real signature* — landed by a signed git commit.

*We are fallible.* This is our best honest belief, not a proof of its own
correctness. Treat it as a falsifiable claim, not gospel.

== The epistemic contract (read before you trust _or_ attack)

The only guarantee is *no intentional overclaim*: where something checks we say
"affirmed"; where it is aspiration, an unbuilt milestone, or a claim we could not
reproduce this session, we say so. An honest claim that later turns out false is
an *error to be fixed* — not a lie. You are invited to bulldoze any claim here:
reproduce (or fail to reproduce) the checks in <<reproduce>>, then tell us where
the discrepancy is.

== What we do NOT claim — remaining red, precisely diagnosed

*Governance: "Trusted-base reduction policy" still fails.* It reports "no
`docs/proof-debt.md` found, but 8 escape hatches exist." We checked: this
repo's *own* Idris2 proofs contain **zero** `believe_me`/`assert_total`/`OWED`
markers — every curated module's own header comment ("no believe_me, no
assert_total") holds under direct `idris2 --check`. The only real
`believe_me`/`OWED` hits found by direct grep are inside vendored
`third_party/gossamer/`. This strongly suggests the checker's "8" figure is
counting vendored code again — the same class of bug as the Java/Kotlin false
positive we did fix (#102) — but we did **not** have the checker's exact
counting script, so we did not fabricate a `docs/proof-debt.md` or guess at the
disposition of hatches we haven't individually verified are even ours. Left
open, precisely diagnosed, for the next session or the owner.

*"Verify Manifest Files" still fails* — for two separable reasons, neither
fixed here: (1) its inline shell script has a genuine bash syntax bug
(`syntax error near unexpected token 'newline'`); (2) even if fixed, it
enforces a per-directory `README.adoc` + numbered `0.N-AI-MANIFEST.a2ml` on
*every* directory in the tree — a convention most of paint-type's source dirs
(`src/paint_core/src/`, `src/backends/cpu/`, etc.) do not and arguably should
not follow. Whether this convention even applies to paint-type is a
`TEMPLATE-APPLICABILITY-POLICY` call for the owner, not something to silently
route around.

*"OSSF Scorecard" still `startup_failure`* and *"Instant Sync" still fails*
(`Parameter token or opts.auth is required`) — the latter needs a repository
secret (a PAT/token) that isn't configured here. Neither is a content bug we
can fix from inside the repo.

*Two divergent `SECURITY.md` files exist* — the substantive, GitHub-native
`.github/SECURITY.md` (406 lines, `paint-type@pm.me` contact — the one CI
checks and the one the fix above targeted) and a stale, 20-line root
`SECURITY.md` with a generic, seemingly-unrelated contact email. Nobody has
reconciled these; not touched here (a content/policy call, not mechanical
hygiene).

*`.machine_readable/STATE.a2ml` is significantly stale.* Dated 2026-05-11,
claims `version = "0.1.0"`, `completion-percentage = 10`, and **0% completion
on v0.2.0 and v0.3.0** — both of which have substantial landed work per
README/ROADMAP history and this session's own verification. Also note:
`.machine_readable/6a2/STATE.a2ml` exists alongside
`.machine_readable/STATE.a2ml` — `6a2` is the deprecated estate naming (the
canonical family is now `descriptiles`; a separate, estate-wide, owner-led
migration, not touched here).

*The `feat/v0.3.0-tools-port` branch is WIP, not evaluated further.* Beyond
confirming it builds and its own tests pass, no review of the `SelectionMask`/
`Clipboard` (MVP-3 selection tools) design or correctness was performed.

*No runnable application exists.* paint.type is pre-alpha; there is nothing an
end user can run today.

*Everything in README "What it will be" remains aspiration* — layers UI,
plugin browser/SDK/sandbox, real-time Burble collaboration, LLM channel,
colour management. None are built.

=== Reproduce it yourself

[source,bash]
----
# Clean rebuild libpt first — a stale zig-out/.zig-cache can produce
# misleading "undefined symbol" link errors that look like a source gap
rm -rf src/interface/ffi/zig-out src/interface/ffi/.zig-cache src/paint_core/target
cd src/interface/ffi && zig build lib && cd -

# Idris2 (0.8.0) — all curated modules; expect all OK
cd src/interface             && idris2 --check Abi/Types.idr Abi/Layout.idr Abi/Foreign.idr
cd verification/proofs/idris2 && idris2 --check ABI/Platform.idr ABI/Compliance.idr Pixel.idr TilePool.idr

# Zig — expect exit 0
zig build && zig build test

# Rust — expect 104/104 passed
cargo test --manifest-path src/paint_core/Cargo.toml
----
