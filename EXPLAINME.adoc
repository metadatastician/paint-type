=== "Linux, macOS, and Windows — same experience, same file format, everywhere"

[quote, README.adoc]
____
* Linux, macOS, and Windows — same experience, same file format, everywhere
____

How this is implemented::
Cross-platform architecture defined in link:TOPOLOGY.adoc[TOPOLOGY.adoc].
Desktop shell (link:https://github.com/hyperpolymath/gossamer[Gossamer]) is designed for multi-platform support.
Native image core (link:https://github.com/hyperpolymath/ephapax[Ephapax]) uses platform-agnostic Rust with Zig FFI bridge.
Build system supports all three platforms via `just` task runner (see link:Justfile[Justfile]).

Caveat::
Not yet tested on all platforms end-to-end. Windows support requires WSL2 or native Zig/C compiler toolchain. macOS and Linux are primary development targets.

=== "The canvas bridge is command-in, dirty-rect-out. Tile data does not cross the bridge as bitmap copies."

[quote, README.adoc]
____
The canvas bridge is command-in, dirty-rect-out. The native core owns the pixel buffer. The webview renders chrome. Tile data does not cross the bridge as bitmap copies — the frontend holds a typed reference into the native core's tile region, verified at compile time by typed-wasm.
____

How this is implemented::
Command-in architecture: link:src/host/src/[src/host/src/] receives commands from webview via JSON-encoded IPC.
Dirty-rect-out: Native core (link:src/paint_core/[src/paint_core/]) computes dirty rectangles and sends to webview for rendering.
Typed references: link:https://github.com/hyperpolymath/typed-wasm[typed-wasm] verifies at compile time that frontend holds typed references into native tile regions.
No bitmap copies: Verified by typed-wasm region schemas in link:src/bridges/[src/bridges/].

Caveat::
Typed-wasm verification requires typed-wasm#127 to be completed upstream. Current draft schemas cannot be compiled yet.
