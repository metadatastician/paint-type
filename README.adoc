== What this is

Paint.NET is one of the best ideas in image editing: capable enough for real work, simple enough to learn without a manual, fast enough to reach for without thinking. Its community knows this. The https://news.ycombinator.com/item?id=39871186[forum threads], the plugin ecosystem, the years of requests — they all say the same thing. The tool is right. The constraints are wrong.

paint.type is the answer to those constraints.

* Open source, community-maintained
* Linux, macOS, and Windows — same experience, same file format, everywhere
* A plugin ecosystem that doesn't live in forum threads
* Real-time collaboration via https://github.com/hyperpolymath/burble[Burble] — built in, not bolted on
* The same simplicity contract Paint.NET made with its users, kept

This is not a GIMP. It is not a Krita. It is not a Photoshop. It is the tool Paint.NET users have been asking for, built by one of them.

== What it will be

=== The core experience

* Layers, blend modes, masks
* Unlimited history — bounded memory, not bounded undo
* Fast everyday editing without professional-pipeline overhead
* Familiar tool set: selection, brush, clone, gradient, shapes, text
* Full colour management — sRGB, Display P3, wide gamut

=== The plugin story, done properly

The Paint.NET plugin ecosystem is one of its best features and one of its biggest frustrations. Finding plugins means searching forum threads. Knowing if a plugin works on your version means hoping. paint.type fixes this:

* In-app plugin browser — find, install, update without leaving the tool
* Signed packages — integrity without gatekeeping
* WASM plugin tier — write once, runs on every platform without recompilation
* Native plugin tier — maximum performance where it matters
* Stable SDK — plugins do not break on every release
* Sandboxed permissions — plugins declare what they need, the runtime enforces it

=== Collaboration

paint.type is Groove-aware. If https://github.com/hyperpolymath/burble[Burble] is running on the same machine, the two find each other automatically. No configuration. No account. No setup. You open a session, someone joins, and you are painting together.

The session includes an LLM channel — not an AI-first experience, not a creative director that tells you what to do, but a participant in the session that holds context, mediates conflicts when two people edit the same region, and answers when addressed. It does not touch the canvas unless explicitly asked. It does not offer unsolicited opinions about your palette.

=== What it will not try to be

paint.type will not try to beat Photoshop at professional print workflows. It will not try to be a full digital content creation suite. It will not accumulate features until it becomes the thing Paint.NET was never supposed to be.

The scope contract is: capable enough for real work, simple enough to reach for without thinking. That contract does not expire.

== Architecture

paint.type is universally cross-platform from moment one. Every architectural decision is constrained by ADR-0002 (link:docs/decisions/0002-foundation-cross-platform.adoc[Foundation — Universal Cross-Platform, Unified Backend/Kernel Surface]).

=== The four-language stack

[cols="1,3"]
|===
| Layer | Choice

| Application language
| **AffineScript** (https://github.com/hyperpolymath/affinescript[hyperpolymath/affinescript]) — algebraic effects separate latency-critical brush traffic from throughput-oriented tile delivery; affine types keep buffer ownership honest.

| Abstract operation surface + proofs
| **Idris2** — defines the `Backend` interface, the capability descriptor, dispatch totality proofs, and cross-language layout proofs. Proofs live here and nowhere else.

| Kernel / FFI layer
| **Zig** — every concrete backend is a Zig module that implements the Idris2-defined C ABI. CUDA, ROCm, Metal, Vulkan, WebGPU, NEON, AVX-512, RVV, FPGA toolchains, audio servers, async I/O, network — all cross into native code here, and only here.

| Unified API surface
| **Zig** — one `paint_api` server exposes the same operation surface over REST, GraphQL, gRPC, Server-Sent Events, and Bebop. Adding a wire protocol is one adapter, not one operation set.

| Portable compilation target
| **typed-wasm** (https://github.com/hyperpolymath/typed-wasm[hyperpolymath/typed-wasm]) — the AffineScript application compiles to a typed-wasm module; backends register as imports verified at the module boundary.

| Canonical distribution artifact
| **OCI multi-arch container** — the buildable, runnable, testable unit from day one. Native binaries are extracted from the container, not parallel build paths.
|===

=== The unified backend/kernel surface

The pattern is taken from https://github.com/hyperpolymath/Axiom.jl[Axiom.jl]: **one abstract operation surface, many concrete backends, capability-driven dispatch, transparent fallback**. Kernel classes (DSP, FPGA, audio, math, GPU, physics, crypto, I/O, vector, tensor) are not directories — they are *capability flags* on a backend. A single backend (`AppleSiliconBackend`, say) reports the kernel classes it can serve and at what precision.

The dispatcher picks the best backend per operation. The reference `CpuBackend` is always available and always correct; accelerated backends are loaded only where the underlying capability is present, and the dispatcher transparently falls back to the reference path if an accelerator does not implement a specific operation.

Topology awareness — NUMA, P/E cores, VRAM, unified memory, PCIe / NVLink / Infinity Fabric / UMA, CXL persistent memory, network reachability cost — is part of the capability report and informs tile placement, batch sizing, and offload decisions.

=== Cross-platform targets

Operating systems::
Linux (every distribution — glibc, musl, Bionic), macOS, Windows, Android, iOS / iPadOS, FreeBSD, OpenBSD, NetBSD, DragonFly BSD, MINIX 3, Illumos / SmartOS, Haiku (best-effort), browsers (via typed-wasm), bare-metal RISC-V via typed-wasm.

Architectures::
x86_64, aarch64, armv7, riscv64, riscv32, ppc64le, wasm32, wasm64. Optional feature tiers: AVX2 / AVX-512 / VAES; NEON / SVE / SME / SME2 / ArmCE; RVV / Zvbb / Zvkn; VSX; Wasm SIMD128 / threads / GC.

*No decision in this repository is permitted to narrow the supported-platform set.* Adding hardware support is opt-in; removing platform support requires an ADR superseding ADR-0002.

== Contributing

paint.type is starting now. The best time to get involved is before the conventions are set.

If you know Paint.NET well — what it gets right, what it gets wrong, what you reach for it for — that knowledge is exactly what this project needs. If you can write Rust, Zig, AffineScript, Ephapax, or web frontend code, there is real work to do. If you want to help define the plugin SDK, the file format spec, or the collaboration model, those conversations are open.

Open an issue, start a discussion, or reach out directly.

`paint-type@pm.me`
