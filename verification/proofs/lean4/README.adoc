= Lean 4 proofs — `verification/proofs/lean4/`

Machine-checked Lean 4 proofs for paint.type's soundness obligations. Both
files are pure core Lean 4 (no Mathlib, no Std), each type-checking with a bare
`lean <file>` invocation. CI re-checks them via `.github/workflows/proofs.yml`
(the `lean4` job), pinned to *Lean 4.13.0*.

[IMPORTANT]
====
*Toolchain pin matters.* `UndoGraph.lean` uses `List.get?`, which was removed
from core Lean after ~4.13 (replaced by `l[i]?` / `getElem?`). Both proofs
check under the pinned *4.13.0*; under a newer Lean (e.g. 4.31.0) `ApiTypes.lean`
still checks but `UndoGraph.lean` does not. Do not bump the pin without porting
`UndoGraph.lean`'s list indexing.
====

== `ApiTypes.lean` — TP-2: public `pt_` API type safety

Models the exported FFI surface (ground truth: `src/interface/ffi/src/main.zig`
— the `pt_*` exports, the `Result` enum `{ok=0, error=1, invalid_param=2,
busy=3}`, opaque `u64` handles, and the null / bad-magic / out-of-bounds
validation discipline) and the `host_core` Command/Response protocol, and proves
the type-safety properties:

* `step_code_valid` / `step_code_in_enum` — every API step returns a `PtCode`
  that is a valid member of the `Result` enum (no out-of-range code).
* `write_ok_implies_live` / `read_ok_implies_live` — an `ok` typed access
  implies the handle was live (the null / bad-magic guard).
* `write_ok_implies_in_bounds` — an `ok` write implies in-bounds coordinates.
* `write_after_free_not_ok` — writing through a freed handle is never `ok`
  (use-after-free is rejected, not serviced).
* `handle_total` — the Command→Response handler is total.
* `handle_shape_correct` / `respShape_lt_four` — protocol response shapes.

== `UndoGraph.lean` — INV-2: undo-graph monotonicity

Proves monotonicity / no-discard properties of the revision graph backing
`paint_core`'s undo history.

NOTE: the use-after-free property (`write_after_free_not_ok`) is proved here at
the model level. It is deliberately *not* re-probed by a runtime test, because
`pt_tile_free` releases the backing allocation — a real post-free access would
be undefined behaviour (libpt's double-free check is a best-effort poisoned-magic
heuristic). The safely-testable discipline (null-handle rejection, bounds
checks, double-free detection) is covered by the libpt unit tests in
`src/interface/ffi/src/main.zig`.
